  - Root cause: `src/modules/gyroscope.js`'s `devicemotion` listener set `gotGyroData = true` unconditionally on every event, even when `event.rotationRate` was `null` (common on devices/browsers that fire motion events with acceleration data only). - **Stale navigation directions piling up in the speech queue** (branch `fix/speech-queue-backlog`, off `develop` — not a numbered spec; fixes a UX lag surfaced during on-device testing of the spec 12/13 CV work: walking through a doorway, the app kept voicing "open the door"/"walk through the doorway" for several seconds after the user was already through, even though the on-screen frame had already updated to the current position):
  - Root cause: two mismatched update policies on the same guidance. The visible frame is last-write-wins — `App.jsx`'s `setLastFrame` always shows the latest capture — but spoken directions go through `speech.js`'s `speechQueue`, a strict FIFO that never drops or supersedes anything queued. Each spoken direction takes longer to say than the ~1s tick interval, so when the user walks faster than utterances drain, directions accumulate and are read out in order, increasingly behind the user's actual position, until the backlog empties. (`STALE_FRAME_MS` doesn't help here: it's checked once when the Claude response arrives, not re-checked when the utterance finally reaches the front of the queue.)
  - `src/modules/speech.js` — `speak()` gained an optional `category` param. A non-interrupt utterance tagged with a category drops any not-yet-started queued utterance of the same category before enqueuing (`speechQueue.filter`), so only the newest pending direction survives — making direction speech last-write-wins like the frame already is. The utterance already handed to `speechSynthesis` (actively playing) is never touched, so it always finishes naturally (no mid-word cutoffs). Utterances with no category (the default) keep the original FIFO behavior unchanged.
  - `src/modules/loop.js` — the three `navigation_direction` `speak()` call sites (scan-win, explore, navigate) pass `category: 'direction'`. Deliberately scoped: hazard/CV alerts (`obstacles.js`/`routeObstacles`, which pass no category and use the `interrupt=true` path for high urgency), rotation warnings, and the "Still scanning"/"Catching up" holdoff cues are all untouched — confirmed CV alerts are unaffected since they never carry a category and so are never filtered by a `'direction'` utterance.

## Architecture Decisions

- High-urgency obstacles fire on the first frame (no 2-frame confirmation) — confirmed correct per PRD R-04/loop latency math; only goal arrival uses 2-frame confirmation.
- `max_tokens` set to 500, not 300, per PRD R-01 — avoids truncated JSON on multi-obstacle frames.
- ~~No CSS files/design system introduced beyond inline styles~~ — **superseded by §11 (Clarity redesign).** The redesign introduces a deliberate visual design: a shared `src/theme.js` tokens module (colors/fonts — not a CSS framework, still consumed via the existing per-component inline-`styles`-object convention), two webfonts loaded in `index.html` (Public Sans display, Atkinson Hyperlegible body — the latter chosen for low-vision legibility), and a light theme. Tap-target (≥56px) and aria requirements from the original decision are still honored.
- **§11 contrast follow-up (open):** the new light palette's *content* text was checked against PRD AC-UI-04/NFR-06 (≥4.5:1) and two failures were fixed (the "DESTINATION" label moved inkFaint→inkMuted; the navigating banner subtitle moved to full white on emerald). White-on-emerald buttons compute to 4.52:1 (pass, narrow margin). Still **pending a full WCAG re-audit on the new palette** (PRD Week-4 "WCAG AA contrast check" task line + AT-CR-05): known borderline item is the "READY" pill (emerald text on emeraldTint ≈3.96:1) — acceptable for now as a small non-content status badge, to be resolved in the contrast audit.
- `stateRef` (goal/context) synced via `useEffect` rather than during render, to satisfy the `react-hooks/refs` lint rule while preserving the "loop always reads fresh state" behavior the spec calls for.
- The WakeLock `visibilitychange` re-acquire listener (Week 2 spec §13) is attached once at module load in `camera.js`, exactly as the spec's snippet shows. Side effect: it will also attempt to silently re-acquire a WakeLock on any tab-visibility change even outside an active navigation session (guarded by `wakeLock === null`, fails silently if no permission context). Not a problem for MVP single-session use; flagged here in case it surprises someone reading the module in isolation.
- Destination extraction (§06 spec) uses a Claude text-only call rather than a regex/keyword stripper — the user explicitly clarified destination phrasing isn't limited to fixed command templates ("take me to X") and may be any direct or indirect description of a place, which only a language-understanding approach can generalize across. This reuses the existing direct-from-browser Claude call pattern (`callClaude`) already in place for vision frames, rather than introducing a new API mechanism or a backend proxy.
- Auto-listen noisy-mic robustness (§08 spec) deliberately does not use a minimum-transcript-length check — the user rejected that approach since length has no relationship to whether a transcript is *correct*. Instead the existing destination-extraction Claude call (§06) was extended with an `ambiguous` flag, so the same call that already resolves the destination also decides whether it needed to guess. This avoids adding a second API call or a separate confidence model just for this check.
- Onboarding (`Onboarding.jsx`) is now shown on every mount (`showOnboarding` defaults to `true`, not gated by `sessionStorage.vg_visited`) and is the sole entry point for auto-listen, because every fresh page load resets the browser's user-activation state — mobile Chrome silently no-ops `speechSynthesis.speak()`/recognition started without a prior tap in that document's lifetime. The previous mount-time auto-listen call (no tap) was the root cause of total silence on-device; the original `vg_visited` skip-onboarding logic was dead code (never wired to `showOnboarding`) and was not revived, since skipping onboarding would reintroduce the silent-audio bug on every fresh load.
- `recognition.start()` runs concurrently with the "Listening for your destination" / "Did you say X?" TTS prompts rather than waiting for them to finish, by deliberate choice — the user wants barge-in (talking over the prompt) rather than a wait-then-listen flow. `startRecognition` (`recognition.js`) takes an optional `onSpeechStart` callback wired to the Web Speech API's `onspeechstart` event, used to call `cancel()` and cut off the prompt the instant speech is detected.
- The no-speech watchdog in `listenForDestination` (`App.jsx`) auto-retries (re-arms the mic) up to `MAX_AUTO_RETRIES` (2) times with "I didn't hear you. Listening again." instead of immediately telling the user to tap the mic — per explicit instruction that silence should re-listen, not dead-end on a manual action. Capped rather than unbounded, so a real hardware/permission failure (not just transient silence) still terminates in the existing "Please type your destination or tap the mic to try again." fallback instead of looping forever. Watchdog timeout is 10s (raised from an initial 4s per explicit instruction).
- The recognition `onError` callback in `listenForDestination` now checks `autoListenSettled` before speaking — the watchdog's `recognitionStopRef.current?.()` calls `recognition.abort()`, which itself fires `onend` → forces `onError` inside `recognition.js` (since nothing had set `settled` yet) → without the guard this produced two spoken fallback messages back to back for a single timeout.
- Guided scan (§09) always turns the user right, in a fixed `ahead → right → behind → left` order, rather than asking which way they'd prefer or alternating — keeps the turn-instruction wording and the final decision's relative-turn math (`normalizeTurnDelta` in `guidedScan.js`) simple and consistent, at the cost of being slightly less natural than e.g. "whichever way is easier." Not user-configurable; revisit only if real-building testing shows users fighting the fixed direction.
- Guided scan's turn-completion detection prefers the gyroscope (`rotationRate.alpha` integrated over time) but falls back to a fixed dwell timer (`SCAN_LEG_NO_GYRO_TIMEOUT_MS`) when `hasGyroData()` is false, rather than blocking indefinitely or requiring a manual confirmation tap — chosen over a manual-confirm flow because it keeps the scan hands-free, and over a pure-timer approach because it's meaningfully more accurate when gyro data is available. Accepted tradeoff: turn angle is approximate (gyro drift, integration noise), not exact 90° — `SCAN_LEG_TURN_TOLERANCE_DEG` (±10°) absorbs some of this, real-building testing will show if it needs widening.
- "Turn around"/"behind" wording is banned from the *navigate* and *explore* prompts (the model has no visual evidence of what's behind the user there) but is allowed in `guidedScan.js`'s own spoken turn instructions — different context: by the time guided scan recommends "turn around," the app has already captured and analyzed a real frame facing that exact direction, so it isn't a guess.
