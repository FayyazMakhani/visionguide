On-Device Computer Vision Layer

01
Purpose & Motivation

Claude vision runs at ~1fps — the API round-trip (capture → encode → POST → inference → response) takes 2–4s at p50. A person walking toward the user can close 1m in under a second. The gap between Claude's 1s polling cadence and real-world obstacle dynamics is a safety latency problem that the existing obstacle module cannot close on its own.

This spec adds MediaPipe Object Detection running at ~15fps on-device as a parallel fast layer. Claude is not replaced: it continues to handle navigation reasoning, goal recognition, and spatial understanding. CV fills the gaps between Claude calls.

Design principle: CV augments Claude, it does not replace it. Claude's role (navigation, goal recognition, spatial reasoning) is unchanged. The raw camera image continues to be sent on every Claude call.

What CV adds

On-device CV~15fpsImmediate safety alerts when a high-risk object is detected between Claude calls

03
Three-Loop Event System

The three loops run at different rates and share data through the atomic cvWorldModel. JavaScript's single-threaded event loop means no true concurrency races exist between synchronous operations across loops — no orchestrator or event bus is needed.

Fast loop   (~15fps, rAF):    cvDetector → cvTracker → cvWorldModel.update()
Medium loop (~3fps,  333ms):  hazardEvaluator reads getSnapshot() → estimateRisk → audio
Slow loop   (~1fps,  1000ms): cvContextBuilder reads getSnapshot() → Claude prompt
Three independent loops at different rates — fast produces, medium and slow consume.

Why these rates

LoopRateRationale
FastrAF (~15fps)Detector + tracker cadence. Matches MediaPipe's optimal VIDEO mode throughput on mobile.
Medium333ms (~3fps)Hazard evaluation does not need to run every detection frame. 3fps catches a fast-approaching person within ~330ms while preventing audio spam.
Slow1000ms (~1fps)Claude API rate — unchanged from existing navigation loop.
04
Data Flow
Camera feed (existing <video> element, shared)
    │
    ▼
cvDetector.js       Raw MediaPipe detections per frame (label, boundingBox, confidence)
    │
    ▼
cvTracker.js        Stable-ID tracked objects (IoU matching, framesVisible, smoothed confidence)
    │
    ▼
cvWorldModel.js     Atomic snapshot — replaced in full each fast-loop tick
    │
    ├──► cvContextBuilder.js  → "CV: person center (high risk)" → Claude prompt (slow loop)
    │
    └──► hazardEvaluator.js   → estimateRisk() → obstacles.js → "person ahead" (medium loop)
Data flows from camera through detector and tracker into a shared world model. Two consumers read independently at their own rates.

Each consumer reads cvWorldModel.getSnapshot() at its own tick rate. The snapshot is either fresh (used) or stale older than CV_CONTEXT_STALENESS_MS (skipped). The fast loop replaces the snapshot atomically — consumers never read partial state.

05
Latency Contract
Explicit contract: Every consumer of cvWorldModel calls getSnapshot() and reads the most recently completed tracker output. No consumer stores a reference to the snapshot across ticks. WorldModel replaces its internal reference atomically (replace-not-mutate) on each fast-loop update.

This guarantees that hazardEvaluator (medium loop) and cvContextBuilder (slow loop) always read from a consistent, fully-formed frame — never a partially updated object. Even if they happen to fire in the same event-loop turn, both see the same stable snapshot from the last completed fast-loop tick.

Staleness guard

Both consumers check snapshot.frameTimestamp against Date.now(). If the snapshot is older than CV_CONTEXT_STALENESS_MS (1000ms), it is skipped entirely. This handles the case where the fast loop pauses (device backgrounded, rAF suspended) without producing stale alerts or prompt noise.

06
New Modules

cvDetector.js

Wraps @mediapipe/tasks-vision ObjectDetector.

CV-DETMediaPipe ObjectDetector wrapper
  • Model: EfficientDet-Lite0 (~4MB). ~15–20fps on iPhone A-series; ~10–15fps on mid-range Android.
  • Running mode: "VIDEO" (continuous, not single-shot). Max 5 results per frame.
  • Confidence threshold: CV_CONFIDENCE_THRESHOLD (0.5) — passed to MediaPipe at init.
  • Shares the existing <video> element from camera.js. No second camera stream.
  • Drives the fast loop via requestAnimationFrame. On each frame, calls cvTracker.update() then cvWorldModel.update().
  • Exposes: init(videoEl), start(), stop(). stop() cancels rAF and closes the detector.

cvTracker.js

IoU centroid matching assigns stable IDs across frames so the world model knows "same object" vs "new object."

CV-TRKIoU centroid tracker
  • On each frame, matches new detections to previous tracked objects by IoU score. A detection with best IoU < 0.3 is treated as a new object and assigned a fresh ascending integer ID.
  • Tracks framesVisible (incremented each matched frame) and smooths confidence as a running average over matched frames.
  • Evicts objects not matched for CV_EVICTION_FRAMES (5) consecutive frames — prevents ghost objects from lingering in the world model.
  • Derives position from bounding box centroid: x < 0.33 → "left", x > 0.67 → "right", else "center".
  • Exposes: update(rawDetections) → trackedObjects[]

cvWorldModel.js

Single atomic snapshot store. Replace-not-mutate discipline — the entire snapshot is replaced on each fast-loop tick, never partially updated.

// Snapshot shape { "objects": [ { "id": 3, "label": "person", "boundingBox": { "originX": 0.2, "originY": 0.3, "width": 0.3, "height": 0.4 }, "confidence": 0.91, "framesVisible": 8, "position": "center" } ], "frameTimestamp": 1234567890 }

Exposes: update(trackedObjects, timestamp) (replaces snapshot), getSnapshot() → snapshot | null.

estimateRisk.js

Pure function. Designed as a seam for future depth model integration — the depthEstimate parameter exists today but is ignored.

function estimateRisk({ label, boundingBox, confidence, framesVisible, depthEstimate = null }) → "high" | "medium" | "low" | "none"
LevelConditions
"none"confidence < CV_RISK_CONFIDENCE_MIN (0.7), or framesVisible ≤ 1 (not yet confirmed across frames)
"low"Any object above confidence minimum not meeting medium/high criteria (small/peripheral objects)
"medium"Large furniture (chair, couch, dining table) in "center" position; or person above confidence minimum but boundingBox.height ≤ CV_PROXIMITY_THRESHOLD
"high"label === "person" AND boundingBox.height > CV_PROXIMITY_THRESHOLD (0.35) AND confidence > CV_RISK_CONFIDENCE_MIN AND framesVisible > 3
Label-aware weighting: A person is higher risk than equivalently-sized furniture regardless of bounding box dimensions. A distant person (small box) still outranks a close potted plant (large box).

hazardEvaluator.js

Runs on the medium loop (every CV_MEDIUM_LOOP_MS, 333ms). Evaluates risk from the world model and routes alerts to the existing obstacles.js speak queue.

CV-HAZHazard evaluation and audio routing
  • Reads cvWorldModel.getSnapshot(). Returns immediately if null or stale (> CV_CONTEXT_STALENESS_MS).
  • Calls estimateRisk() on each tracked object.
  • Fires audio alert via existing obstacles.js routeObstacles() at HIGH priority when ALL three conditions hold:
    • estimateRisk() returns "high"
    • Object's label class has not been alerted in the last CV_ALERT_COOLDOWN_MS (3000ms)
    • Speech module's last utterance ended > 2000ms ago — deference to active Claude navigation guidance
  • Alert text is brief and standardized (not Claude-generated): "person ahead", "obstacle ahead" — derived from label and position.
  • Exposes: start(), stop()

cvContextBuilder.js

Called once per slow-loop Claude tick from loop.js. Produces a compact, filtered summary of current scene state for Claude.

CV-CTXClaude prompt context builder
  • Reads cvWorldModel.getSnapshot(). Returns null if null or stale.
  • Filters to objects with estimateRisk() result of "medium" or "high" only, max 3 objects. Low-risk and unconfirmed objects are excluded — they add noise without aiding navigation.
  • Returns compact English string: "CV: person center (high risk), chair right (medium risk)"
  • Returns null when no qualifying objects. Claude prompt is sent without any CV prefix — no token cost when CV has nothing useful to add.
  • Exposes: build() → string | null

09
Acceptance Tests

All AT-CV tests run on a physical device (iPhone iOS Safari or Chrome).

  • AT-CV-01 — After Start, browser DevTools → Console shows a "CV detector ready" log within 5 seconds of navigation beginning.
  • AT-CV-02 — Hold phone toward a person. Within ~1s, a brief spoken alert fires ("person ahead") without waiting for the next Claude response. Confirms medium loop fires independently of the slow Claude loop.
  • AT-CV-03 — Walk toward a chair in an otherwise clear corridor. The Claude API request visible in DevTools → Network includes a "CV: chair..." prefix line in the request body. Confirms context builder enriches Claude prompts when qualifying objects are present.
  • AT-CV-04 — Point phone at an empty hallway with no obstacles. No CV audio alert fires. Confirms alert suppression when no high-risk objects are present.
  • AT-CV-05 — Press Stop. No further CV audio fires after Stop. DevTools confirms no further rAF callbacks from cvDetector. Confirms clean teardown.
  • AT-CV-06 — While Claude is speaking a navigation direction, a CV alert for an object that appeared mid-sentence does not interrupt until speech ends. Confirms speech-deference logic (2000ms post-utterance holdoff).
  • AT-CV-07 — Point phone steadily at a person for 10 seconds. Alert fires ≤ 3 times (once per CV_ALERT_COOLDOWN_MS, 3s). Confirms per-class cooldown.

Definition of Done
  • npm install, npm run build, and npm run lint all pass clean.
  • AT-CV-01 through AT-CV-07 pass on a physical device (iPhone iOS Safari or Chrome).
  • Existing acceptance tests AT-01–AT-W4-12, AT-SC-01–07, AT-EX-01–06 still pass — navigation, obstacle detection, and arrival behavior are unchanged.
  • context/progress-tracker.md and context/specifications/visionguide-prd.html updated to reflect this spec, per this repo's existing process.
Future extensibility: Once this spec ships, the estimateRisk() function's depthEstimate seam and the cvContextBuilder's JSON-first world model are the designated integration points for depth estimation and fine-tuned model variants respectively — no architecture changes needed.
