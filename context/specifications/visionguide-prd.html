Problem Statement

GPS navigation has transformed outdoor travel for visually impaired individuals, but GPS signals are unavailable or unreliable indoors. Complex indoor environments — hospitals, transit hubs, shopping malls, airports, apartment buildings — present significant and daily navigation challenges.

Existing indoor solutions require specialized building infrastructure: Bluetooth beacons, NFC tags, pre-mapped floor plans. This makes them expensive to deploy, limited to specific buildings, and unavailable in the vast majority of real environments people navigate every day.

The core problem is: a visually impaired person in an unfamiliar indoor space has no reliable, independent way to locate a door, avoid an obstacle, or follow directional signage without assistance from another person.

Product Vision

VisionGuide turns a standard Android smartphone into an indoor navigation assistant using only its camera and an AI vision model. No specialized hardware. No pre-installed building infrastructure. No internet-dependent map database.

Vision statement: Any visually impaired person should be able to pick up a phone running VisionGuide, state where they want to go, and receive real-time spoken guidance to navigate there independently — in any indoor environment, on their first visit.

Safety boundaries

VisionGuide is a supplemental navigation aid only. It does not replace a white cane, guide dog, or other primary mobility aid. The system makes probabilistic assessments from a camera feed and will make errors — including hallucinating clear paths that are not clear. Users must maintain use of their primary mobility aid at all times during navigation. This boundary must appear in the onboarding voice prompt at every session start and in all public-facing product descriptions.

The system operates at ~1 fps with 3–5 second processing latency. At normal walking speed (~1.4 m/s), the user may travel 4–7 meters between a frame being captured and the resulting audio playing. VisionGuide is designed for deliberate, cautious movement — not normal walking pace.

Design principles

PrincipleWhat it means in practice
DP-01Audio firstEvery interaction and feedback is spoken. The screen is secondary. No interaction requires looking at the phone.
DP-02Interrupt for dangerHigh-urgency obstacles always interrupt ongoing speech. Latency for danger alerts is prioritized above all else.
DP-03Zero infrastructureWorks in any building. No beacons, no floor plans, no pre-registration required.
DP-04Minimal UIOne button to start. One input for the goal. Nothing the user needs to tap during navigation.

Functional Requirements

Camera & Vision

FR-CAM-01 Camera stream initialization MVP
The app must request camera access via getUserMedia({ video: { facingMode: 'environment' } }) on session start. If permission is denied, it must speak an error and display recovery instructions.
  • Use rear-facing camera only
  • Minimum resolution: 640×480
  • Stream must be rendered in a hidden <video> element to keep it alive
FR-CAM-02 Frame extraction at 1 fps MVP
Every 1000ms, the app draws the current video frame to a canvas element and exports it as a JPEG at 70% quality. The resulting base64 string is passed to the inference pipeline.
  • Frame must be dropped if a prior API call is still pending
  • API response must be discarded if the frame was captured more than 4500ms ago (p95 latency + 500ms buffer — responses older than this reflect a scene the user has moved significantly beyond)
  • Canvas dimensions: 640×480 (fixed, not dynamic)
FR-CAM-03 Screen wake lock MVP
The app must request a WakeLock on session start to prevent the screen from sleeping during navigation. If WakeLock is unavailable, it must warn the user via TTS that the screen must remain on.

Navigation

FR-NAV-01 Goal input MVP
On page load, the app automatically starts listening for a plain-language destination (e.g. "the elevator", "room 204", "the exit") — see FR-NAV-06. The microphone button remains available afterward to retry or change the destination by voice; the text field remains the fallback/manual path at all times. The goal string is stored in app state and included in every API call for the session duration.
  • Goal is immutable per session. To change destination, user restarts.
  • Manual voice input (mic button): tap microphone button, speak goal, recognition result populates the text field for confirmation before starting navigation
  • Text field must have aria-label="Navigation destination"
  • Field must be focused on page load for screen reader compatibility
  • If Speech Recognition is unavailable, fall back silently to text-only input
FR-NAV-04 Destination extraction from arbitrary phrasing MVP
The raw goal string from FR-NAV-01 is not limited to a fixed command template ("take me to X", "go to X") — it may be a direct command, a question, or an indirect description of a need (e.g. "I need to wash my hands", "there's a meeting in room 204, get me there"). Before navigation starts, the app resolves this raw input to a clean destination noun phrase via a single Claude text-only call, which also returns whether the input was ambiguous (see FR-NAV-06).
  • Runs once, when Start is tapped (manual path) or once a voice transcript is captured (auto-listen path, FR-NAV-06) — not on every keystroke or voice result. The text field continues to display the user's raw input until then.
  • If the extraction call fails (network/API error) or returns nothing usable, the app falls back to the raw input string, treats it as unambiguous, and navigation still proceeds.
  • The resolved destination replaces the goal string used in all subsequent navigation speech and API calls (FR-NAV-02, FR-NAV-03, FR-SCAN-01–05) for the rest of the session.
  • The manual Start-button path ignores the ambiguity flag entirely — no spoken confirmation is added there. Only the auto-listen path (FR-NAV-06) acts on it.
FR-NAV-02 Navigation directions MVP
Each API response includes a navigation_direction string of maximum 8-10 words (tightened from 15 — shorter utterances reduce speech-queue backlog and the window during which displayed text can drift ahead of audio). This is spoken via TTS after any queued speech completes, unless an obstacle interrupt is pending.
  • Directions must use compass-free language: "turn left", "straight ahead", "move right"
  • Same direction must not be spoken twice consecutively unless >10 seconds have elapsed
  • Directions must never say "behind" or instruct the user to turn around — the camera only sees forward, so there is no visual evidence for anything behind the user. If the path ahead is blocked, guidance must pick left or right based on which side of the current frame shows an opening, or state plainly that there's no clear path, rather than guess (see FR-NAV-07).
FR-NAV-07 Grounded direction guidance MVP
Two mitigations for direction guidance that isn't grounded in what the current frame actually shows, applied to the navigate (FR-NAV-02), explore (FR-SCAN-04), and guided-scan (FR-SCAN-06) prompts alike:
  • No unseen-direction guessing: the model is never told it may say "behind"/instruct a turn-around — only left/right/ahead, which the current frame can actually support. When the path ahead is blocked, it must choose left or right toward a visible opening in that same frame, or state explicitly that no clear path is visible.
  • Left/right self-check: before stating left or right (for a navigation direction or an obstacle's reported side), the model is instructed to locate the object in the frame first — left half of the image is the user's left, right half is the user's right, the image is not mirrored — and state the matching side. This is a mitigation for a known vision-model spatial-reasoning weakness (mirrored left/right reports), not a guaranteed fix; see prompt-tuning-log.md's "Wrong direction" failure category for ongoing tracking.
FR-NAV-03 Goal arrival announcement MVP
When goal_found: true is returned with goal_confidence ≥ 0.8 in two consecutive frames, the app must speak "You have arrived at [goal]" and stop the navigation loop.
  • Two-frame confirmation required to prevent false positives — an unbroken streak of two qualifying frames; any frame that fails either condition resets the counter
  • Loop must fully stop — no further API calls after arrival
FR-NAV-05 Destination memory when out of view MVP
Once the destination has been seen with goal_confidence at or above the scan/explore threshold (0.5), the app remembers the direction it was last seen in. If a later frame doesn't find the destination, that last-known direction is included as a hint in the next API call so guidance can route the user back toward it instead of treating the destination as lost.
  • A sighting expires after 30 seconds — old enough to survive a brief look-away, not so old it guides the user toward a now-irrelevant direction if they've moved on
  • Memory is cleared at the start of every new navigation session
  • This does not change arrival confirmation (FR-NAV-03) — the hint only affects guidance while the destination is out of frame, not whether arrival is confirmed
FR-NAV-06 Auto-listen for destination on launch MVP
Rather than requiring a mic tap or a dismissed onboarding modal first, the app starts listening for a destination automatically on page load, speaking "Listening for your destination." No onboarding modal is shown.
  • If SpeechRecognition is unavailable in this browser, no auto-listen attempt is made; the app speaks "Voice recognition is not available. Please type your destination." and the user proceeds via the manual text field + Start button (FR-NAV-01).
  • On a captured transcript, the app runs destination extraction (FR-NAV-04). If the result is not flagged ambiguous, the app speaks "Heading to [destination]." and immediately begins navigation (camera init, safety disclaimer per FR-UI-03 if due, then FR-SCAN-01) — no manual Start tap required.
  • If the result is flagged ambiguous (transcript was noisy, fragmentary, or otherwise required heavy guessing to resolve), the app instead asks "Did you say [destination]? Say yes or no." and listens for a yes/no answer before proceeding. A "yes" (or any answer not containing "yes") branches to the same "Heading to..." + auto-start, or to "Okay, let's try again." followed by re-entering auto-listen, respectively.
  • On no speech / recognition error during either the initial listen or the yes/no confirmation listen, the app speaks a fallback ("Didn't catch that..." or "Please type your destination or tap the mic to try again.") and returns to an idle state where the manual mic button and text field remain available.
  • This is a one-time automatic attempt on mount. After it resolves (successfully or not), all further voice input goes through the existing manual mic button (FR-UI-03), unchanged.

Scan & Explore Phases

FR-SCAN-01 Guided 4-direction scan on Start MVP
The phone may not be pointed at the destination when navigation starts, and a free, unguided rotation gave the user no way to tell which way to start walking once it ended. Every Start tap now enters a guided scan phase: the app instructs the user to turn roughly 90° right and stop, four times in a row (covering ahead, right, behind, and left relative to where they started), analyzing one still frame per stop instead of a continuous rotating sweep (superseded: scan no longer requires continuous slow rotation; supersedes the previous free-rotation behavior of this requirement).
  • Frame capture only happens once per stop (after the turn completes and a brief settle pause), not continuously — see FR-SCAN-02 for turn-completion detection
  • Uses a dedicated per-leg scan prompt distinct from the navigation prompt, which also returns a path_openness score (0.0–1.0) for that direction
  • Exits to navigation phase immediately, skipping remaining stops, once any stop's frame returns a non-null navigation_direction with goal_confidence at or above 0.5 — same threshold as before
  • If no stop finds the destination, see FR-SCAN-06 for what happens after all four are analyzed
  • The first stop's instruction must enqueue in SpeechQueue after the mandatory safety prompt (FR-UI-03), never interrupting it
  • Same stale-frame guard as navigate phase (R-01): a response is dropped (no phase transition, no spoken direction, leg recorded as unscored) if its frame is older than 4500ms when the response arrives — obstacles in that frame are still routed
FR-SCAN-02 Gyroscope-gated turn detection during scan MVP
The app tracks yaw rate and cumulative turn angle via DeviceMotionEvent to know when each scan stop's ~90° turn is complete, and still warns "A little slower." (no more than once per 2.5s) if rotation exceeds 30°/s — fast rotation both blurs the eventual frame and degrades the turn-angle estimate.
  • A stop's turn is considered complete once accumulated rotation is within 10° of the 90° target, or — if no rotation-rate data is available at all (denied permission, unsupported browser) — once a fixed 2.5s fallback dwell has elapsed for that stop, so scan phase can never hang waiting on a sensor that isn't there
  • "No rotation-rate data available" is determined per devicemotion event, not per session: a device that fires motion events containing only acceleration data (rotation rate absent on that event) is treated as having no usable rotation data for the fallback check above, even though motion events are arriving. Bug fix: an earlier implementation treated the mere arrival of any devicemotion event as proof of usable rotation data, which on such devices permanently disabled the 2.5s fallback while the turn angle itself never accumulated — stalling a stop indefinitely regardless of how much the user actually turned.
  • A hard 6-second ceiling applies to every stop's turn-completion check regardless of rotation-rate availability — a defensive backstop in case rotation-rate data is present but doesn't track the turn as expected for how the phone is actually held (chest-height, pointing forward), so a stop can never stall longer than this even if the angle-based and no-gyro-fallback checks both fail to fire
  • On iOS 13+, an explicit DeviceMotionEvent.requestPermission() prompt is requested when starting the loop (inside the same Start-tap gesture), since rotation events otherwise never fire there at all
  • Gyroscope use is purely relative (rotation rate integrated over time) — never an absolute/magnetometer heading, and never spoken to the user as a compass direction
  • A brief settle pause (~600ms) after each detected stop, before capturing, avoids analyzing a motion-blurred frame
  • Gating applies only during scan phase. Explore and navigate phases never block on rotation speed.
FR-SCAN-03 Scan timeout into explore phase MVP
The 4-stop scan must not be able to hang indefinitely (e.g. a stop's turn never registering as complete). If 20 seconds elapse in scan phase without completing the sequence, the app falls back to the same generic explore announcement described in FR-SCAN-06's "no clear direction" case and switches the capture interval to 1000ms (1 fps), since the user is now walking rather than standing still.
FR-SCAN-04 Explore phase guidance MVP
In explore phase, Claude is asked to look for the destination first; if it isn't visible, it instead guides the user toward open navigable space (hallways, corridors) or directional signage so the user keeps moving through the building rather than stalling.
  • goal_confidence distinguishes a true destination match (≥0.5, transitions to navigate phase) from general walking guidance toward navigable space (direction returned at confidence 0.0)
  • Obstacle alerts continue to fire during explore phase using the existing ObstacleRouter behavior (FR-OBS-01)
  • The app must never go silent on a non-stale explore frame: if Claude returns no navigation_direction (e.g. a fully blocked view), the app speaks a fallback ("I don't see a clear path. Try turning left or right.") instead of saying nothing
  • Subject to the same grounded-direction rules as navigate phase — see FR-NAV-07
FR-SCAN-05 Explore timeout failure MVP
If 90 seconds elapse in explore phase without a destination match, the app speaks "I wasn't able to find [goal]. Please ask someone nearby for help." and performs a full Stop. The next Start always begins again in scan phase.
FR-SCAN-06 Guided scan direction hand-off MVP
If none of the 4 scan stops find the destination directly (FR-SCAN-01), the app must not hand off to explore phase with no real direction — the previous free-rotation behavior's generic "follow my directions" message gave the user nothing concrete to act on, especially if the phone happened to end the scan facing a wall.
  • The stop with the highest path_openness score is chosen, unless any of its obstacles is urgency: "high" (treated as fully blocked, score 0)
  • If the best stop's score is at or above 0.4, the app speaks a concrete turn instruction toward it before entering explore phase — e.g. "Turn right, I saw an open hallway that way," "Turn around, there's a clearer path behind you," or "Stay facing this way" if the best stop is the direction already faced. The turn is computed relative to the user's current facing (after the 4th stop), not as a compass heading.
  • If no stop's score reaches 0.4 (all roughly equally blocked or open), the app falls back to the same generic explore announcement as a scan timeout (FR-SCAN-03) rather than guessing further

Obstacle Detection

FR-OBS-01 Obstacle urgency tiers MVP
The API returns obstacles as an array, each with a urgency field: high, medium, or low. Routing logic:
  • High: Cancel current speech and speak obstacle alert immediately on the first frame. No confirmation delay — at 1 fps pipeline latency, introducing a 2-frame wait consumes the entire available safety window. Accept higher false positive rate as the correct tradeoff.
  • Medium: Queue alert after current direction finishes speaking.
  • Low: Log only. Do not speak.
FR-OBS-02 Obstacle alert language MVP
Obstacle alerts must be short, directional, and action-oriented. Examples: "Stop — step ahead", "Chair on your left", "Door swinging open ahead". Maximum 8 words. Must not use "I see" or "it looks like".

Audio Output

FR-AUD-01 TTS via Web Speech API MVP
All spoken output uses window.speechSynthesis. Voice selection: prefer the first available en-US voice with localService: true. Fall back to default if unavailable.
  • Speech rate: 1.15 (slightly faster than default for navigation use; raised from the original 1.05 to shorten how long each utterance occupies the speech queue)
  • Pitch: 1.0 (no modification)
  • Volume: 1.0
FR-AUD-02 Silence fallback MVP
If no API response is received within 3 seconds of a frame being sent, the app speaks "Still scanning" and continues the loop. This prevents unexplained silence. Maximum one "Still scanning" per 10 seconds.
FR-AUD-03 Alert spacing and text/speech sync MVP
Two SpeechQueue behaviors are required to keep spoken output from feeling rushed or desynchronized from the on-screen text:
  • When an obstacle interrupt (FR-OBS-01 high urgency) or the arrival announcement (FR-NAV-03) cuts off speech that is actively playing, the new utterance starts after the same brief pause a natural queue transition would get, not instantly — an instant cut-in is what makes back-to-back alerts feel like they collide. If nothing is currently playing, the new utterance still starts immediately.
  • The on-screen status text must update at the moment its matching utterance actually starts playing, not at the moment it's enqueued — queued/delayed speech otherwise lets the displayed text race ahead of the audio.

UI & Controls

FR-UI-01 Minimal single-screen UI MVP
The UI consists of: goal text input, microphone button for voice goal entry, Start/Stop button, a status indicator (scanning / navigating / arrived), and a last-spoken text display. No other interactive elements during navigation.
  • All tap targets minimum 56px height
  • High contrast: text on background must meet WCAG AA (4.5:1)
  • Font size: minimum 18px body, 24px for status
  • Microphone button must have aria-label="Speak your destination"
FR-UI-02 Screen reader compatibility MVP
The app must be fully operable via TalkBack on Android Chrome without sighted assistance. All interactive elements must have appropriate ARIA labels. Status changes must be announced via aria-live="assertive" regions.
FR-UI-03 Voice goal input via Web Speech Recognition MVP
A microphone button on the pre-navigation screen activates SpeechRecognition (Chrome Web Speech API) for manual voice goal entry/retry. The recognized text populates the goal field. The user confirms by tapping Start. (The automatic listen-on-launch attempt, FR-NAV-06, uses the same underlying recognition module but has its own confirmation/auto-start behavior, distinct from this manual mic-button flow.)
  • Use webkitSpeechRecognition with lang: 'en-US', interimResults: false, maxAlternatives: 1
  • While listening, speak "Listening..." and show visual indicator
  • On recognition result, populate text field and speak back the recognized text: "I heard: [goal]. Tap Start to begin."
  • On recognition error or no-result, speak "Didn't catch that. Please try again or type your destination."
  • If SpeechRecognition is unavailable (non-Chrome browser), hide mic button and use text-only input silently
  • Safety prompt (mandatory): When navigation begins (via Start tap, or via the auto-listen flow's FR-NAV-06 auto-start), before the navigation loop begins, speak: "VisionGuide is a navigation aid only. Keep using your cane or other mobility aid." Plays once per calendar day, on the first navigation start of that day; cannot be skipped on that first play. Subsequent starts the same day proceed directly to the scan instruction (see FR-SCAN-01) without replaying it.
FR-UI-04 Analyzed-frame preview for sighted observers MVP
The last-spoken text display additionally shows a still image of the exact camera frame that was analyzed to produce the instruction currently on screen — letting a sighted companion, demo audience, or developer see what the AI actually saw. This is a visual-only aid for sighted observers; the app's primary blind/low-vision user gets no benefit from it and it is not announced via screen reader.
  • The image shown is the frame behind the current spoken navigation_direction — ticks that don't produce a spoken direction (e.g. "Still scanning", "Turn right and stop.") leave the previously shown image in place rather than clearing or flickering it.
  • During guided scan phase (FR-SCAN-01), the image additionally updates after every stop's capture, win or lose — not just on an early goal-found exit — so a sighted observer can confirm a fresh frame was actually analyzed at each of the 4 stops, even though most stops have no spoken direction of their own.
  • Not part of the aria-live region used for TalkBack announcements (FR-UI-02) — an image conveys nothing to that audience.
  • Cleared on Stop and at the start of a new navigation session, so no frame from a prior session/destination is ever shown alongside a new one.

System Architecture

VisionGuide is a two-layer system: a browser client and the Anthropic Claude Vision API. The browser calls Anthropic directly using a key stored in a .env file. There is no proxy, no server, and no deployment infrastructure beyond static file hosting.

Figure 1 — System architecture: the browser layer (camera, canvas, React app, parallel on-device MediaPipe CV, Web Speech) calls Anthropic directly. The camera stream feeds both the 1 fps Claude path and the ~15 fps on-device CV layer (Spec 12).

API Contract

Claude Vision prompt (system)

// System prompt — navigation phase. Source of truth: src/prompts/system.js // (buildSystemPrompt). Scan, explore, guided-scan, and destination-extraction // each use their own phase prompt in the same file. This shows the navigate // schema and core rules, abbreviated. You are an indoor navigation assistant for visually impaired users. Analyze the image and return ONLY valid JSON. No preamble, no markdown, no explanation. JSON structure: { "obstacles": [ { "type": string, "direction": string, // "left" | "center" | "right" "urgency": "high" | "medium" | "low" } ], "navigation_direction": string, // max 8-10 words; null when path_blocked "path_blocked": boolean, "goal_found": boolean, "goal_confidence": number // 0.0 to 1.0 } // Phase-specific additions: explore adds "small_space": boolean; guided-scan adds // "path_openness": number (0.0-1.0); describe mode (deferred, S16) adds "scene_description". Rules (abbreviated — see src/prompts/system.js for the authoritative full text): - navigation_direction: max 8-10 words, action-oriented, naming a specific visible landmark. - Spatial words only: left, right, ahead. Never "behind" or "turn around" (the camera sees forward only, so there is no evidence of what is behind the user). Never compass directions. - Left/right are the camera-holder's perspective; locate the object in the frame first (left half of the image = user's left). The image is not mirrored — do not swap sides. - path_blocked=true for a wall, dead end, or enclosed space with no passable gap; then navigation_direction MUST be null. Never invent a path over a wall or through a closed, unmarked door. - urgency=high: stationary object directly blocking the path within 1 metre only. Passing people are urgency=medium at most. - goal_found=true only when the destination is immediately at hand (reachable), not when a matching sign is merely legible from down a hallway. - Do NOT include scene descriptions or commentary. Return valid JSON only. Nothing else.

Component Specifications

CameraStream

Initializes getUserMedia, renders stream to <video>, requests WakeLock. Exposes getFrame(): string which draws to canvas and returns base64 JPEG.

NavigationLoop

Owns the setInterval and the scan → explore → navigate phase state machine (FR-SCAN-01–06). Phase determines the capture interval (500ms scan, 1000ms explore/navigate), which prompt is sent, whether gyroscope gating applies (scan only), and the confidence threshold for transitioning toward the destination. During scan phase, frame capture is additionally gated by GuidedScan's per-stop turn/settle staging — no frame is captured until a stop's turn is complete and the brief settle dwell has elapsed. Checks pending flag before calling. Calls CameraStream.getFrame(), assembles payload, POSTs directly to https://api.anthropic.com/v1/messages with the key from import.meta.env.VITE_ANTHROPIC_API_KEY, sets pending = false on resolution. Drops the response if the frame was captured more than 4500ms ago (STALE_FRAME_MS, matching FR-CAM-02 / R-01). Phase and GuidedScan's stop sequence both reset on every Stop.

Gyroscope

Singleton wrapping DeviceMotionEvent, used only during scan phase. Initialized inside the Start tap handler, not on page load — also requests iOS's DeviceMotionEvent.requestPermission() there, in the same gesture context, since iOS 13+ otherwise never fires rotation events at all. Exposes isRotatingTooFast() (true above 30°/s yaw rate; always false if DeviceMotionEvent is unavailable, so scan phase is never blocked on unsupported devices) and shouldWarnRotationSpeed() (debounced to at most once per 2.5 seconds). Also integrates signed rotation rate over time into a resettable accumulator (resetTurnAccumulator() / getAccumulatedTurnDegrees()) used by GuidedScan to detect when a stop's ~90° turn is complete, plus hasGyroData() so callers can fall back to a timer when no rotation data is available at all — only set once a devicemotion event actually carries rotation-rate data, not merely once any motion event has fired (FR-SCAN-02).

GuidedScan

Owns the 4-stop scan sequence (FR-SCAN-01/02/06): ahead → right → behind → left, each reached by a ~90° turn detected via Gyroscope (or a fixed dwell-time fallback, or — regardless of Gyroscope's reported state — a hard 6-second per-stop ceiling, FR-SCAN-02). Stages each stop through awaiting_turn → settling → ready before NavigationLoop captures and analyzes a frame, and reports the captured frame back to the UI preview (FR-UI-04) at the end of every stop, not only on a goal-found short-circuit. Records each stop's path_openness/obstacles/navigation_direction, and after all 4 are recorded (with no earlier goal-found short-circuit), decide() picks the highest-scoring open, unblocked stop and returns a relative turn instruction toward it, or reports no clear direction if none clears the minimum openness bar.

ObstacleRouter

Receives parsed JSON. On urgency: high, cancels current speech and speaks the alert on the first frame — no 2-frame confirmation, since at 1 fps plus pipeline latency a 2-frame wait consumes the entire available safety window (FR-OBS-01, R-04). Medium urgency enqueues after the current direction finishes. Low urgency discards.

SpeechQueue

Wraps speechSynthesis. Exposes speak(text, interrupt = false). When interrupt = true, calls speechSynthesis.cancel() first. Tracks last spoken text and timestamp to enforce deduplication (no repeat within 10s).

GoalTracker

Maintains consecutiveGoalFoundCount. Increments on goal_found: true && goal_confidence ≥ 0.8. When count reaches 2, fires arrival event and stops NavigationLoop.

Acceptance Criteria

All acceptance criteria below must pass before the build is considered demo-ready. Each maps to a functional requirement.

Camera & Vision

  • AC-CAM-01: On page load, browser prompts for camera permission. Granting permission starts video stream within 2 seconds.
  • AC-CAM-02: Denying camera permission causes app to speak "Camera access is required. Please allow camera in browser settings." and display written instructions.
  • AC-CAM-03: Frame is captured at 640×480 JPEG, base64-encoded, and included in an Anthropic API request within 100ms of interval tick.
  • AC-CAM-04: If a prior API call is pending, frame is silently dropped and no new request is made.
  • AC-CAM-05: WakeLock is acquired on session start. Screen does not sleep during a 10-minute active session.

Navigation

  • AC-NAV-01: User types a goal and taps Start. The goal string is included in every subsequent API call for the session.
  • AC-NAV-02: Navigation direction is spoken within 3 seconds (p50) of frame capture.
  • AC-NAV-03: The same direction is not spoken twice within 10 seconds.
  • AC-NAV-04: When goal_found=true and goal_confidence≥0.8 in two consecutive frames, app speaks "You have arrived at [goal]" and stops the loop.
  • AC-NAV-05: After arrival, no further API calls are made until the user restarts.
  • AC-NAV-06: Typing or speaking an indirect/descriptive phrase (e.g. "I need to wash my hands") or a full command phrase (e.g. "take me to the bathroom") resolves to a clean destination noun phrase before any navigation speech is generated; if the extraction call fails, the raw input is used instead and navigation still proceeds.

Scan & Explore Phases

  • AC-SCAN-01: Tapping Start always plays the "hold still, facing forward" instruction before any navigation direction is spoken. On the first Start of the calendar day, the safety prompt plays first; on later Starts that same day, the scan instruction plays directly without the safety prompt repeating.
  • AC-SCAN-02: Rotating the phone faster than 30°/s during a scan-phase turn triggers "A little slower.", spoken no more than once per 2.5 seconds, without preventing turn-completion detection from still progressing.
  • AC-SCAN-03: If the destination or a relevant sign is visible at any of the 4 scan stops, the app speaks the navigation direction for that stop and switches to navigate phase immediately, skipping any remaining stops.
  • AC-SCAN-04: An obstacle detected at any scan stop fires the existing obstacle alert without exiting scan phase.
  • AC-SCAN-05: If the 4-stop scan sequence does not complete within 20 seconds (e.g. a turn never registers as done), the app automatically begins explore phase with the generic hand-off message, without user input and without stopping.
  • AC-SCAN-06: During explore phase, the app gives walking guidance toward open space or signage at 1 fps — never staying silent on a non-stale frame, even when Claude returns no direction — and transitions to navigate phase as soon as the destination becomes visible.
  • AC-SCAN-07: If explore phase exceeds 90 seconds without a destination match, the app speaks the failure message and performs a full Stop.
  • AC-SCAN-08: Tapping Stop during scan or explore phase halts the loop cleanly with no further speech; the next Start always begins again at scan stop 1 ("ahead").
  • AC-SCAN-09: After all 4 scan stops complete without finding the destination, if one stop scored clearly more open/navigable than the others, the app speaks a concrete turn instruction toward that direction (e.g. "Turn right, I saw an open hallway that way") before entering explore phase, rather than a direction-less generic message.
  • AC-SCAN-10: If all 4 scan stops score below the minimum openness bar (or are blocked by a high-urgency obstacle), the app falls back to the same generic explore hand-off message used on a scan timeout, rather than guessing a direction.
  • AC-SCAN-11: On a device/browser with no usable rotation-rate data (denied iOS permission, no DeviceMotionEvent support, or motion events that only ever carry acceleration data), each scan stop's turn still completes via the fixed fallback dwell timer instead of hanging indefinitely.
  • AC-SCAN-13: Even if rotation-rate data is present but never accumulates toward the 90° target despite genuine turning, a scan stop still completes within 6 seconds via the hard ceiling, rather than stalling toward the 20-second whole-scan timeout (AC-SCAN-05) on every stop.
  • AC-SCAN-12: Navigate- and explore-phase guidance never says "behind" or instructs the user to turn around; at a dead end it picks left/right toward a visible opening or states there's no clear path instead.

Obstacle Detection

  • AC-OBS-01: A high-urgency obstacle returned in two consecutive frames causes current speech to be cancelled and the obstacle alert to be spoken immediately.
  • AC-OBS-02: A medium-urgency obstacle is spoken after current navigation direction completes, not interrupting it.
  • AC-OBS-03: A single high-urgency frame (not confirmed by a second) does not trigger an interrupt.
  • AC-OBS-04: Obstacle alert text is 8 words or fewer and contains a direction word.

Audio

  • AC-AUD-01: If no API response arrives within 3 seconds of a frame being sent, app speaks "Still scanning".
  • AC-AUD-02: "Still scanning" is not spoken more than once per 10 seconds.
  • AC-AUD-03: All spoken text is audible at phone default volume in a moderately noisy environment.

UI & Accessibility

  • AC-UI-01: All tap targets are at minimum 56×56px.
  • AC-UI-02: Goal input field is focused on page load when using TalkBack.
  • AC-UI-03: Status changes ("Scanning", "Arrived") are announced by TalkBack without user interaction.
  • AC-UI-04: All text meets WCAG AA contrast ratio (4.5:1 minimum).
  • AC-UI-05: App is fully operable via TalkBack without sighted assistance (verified by manual test).
  • AC-UI-06: The displayed analyzed-frame image (FR-UI-04) updates in step with each new spoken navigation instruction, is cleared on Stop, and shows no stale frame at the start of a new session.
  • AC-UI-07: During guided scan phase, the displayed analyzed-frame image (FR-UI-04) updates after every one of the 4 stops' captures, not only on an early goal-found exit — confirming a fresh frame was analyzed at each stop even when that stop has no spoken direction.

Reliability

  • AC-REL-01: App runs without JS error or silent failure for a 10-minute session.
  • AC-REL-02: On API error (non-200 response), app speaks "Still scanning" and resumes the loop on the next tick.

Deferred but Architected

These features are not built in the MVP but must be accounted for in code structure now. Adding them later must not require a rewrite.

Scene Description Mode

When active, Claude returns a scene_description field describing the general environment in one sentence. This is spoken in addition to (or instead of) the navigation direction, depending on mode.

Architecture requirement: The system prompt must be assembled by a buildSystemPrompt(mode) function, not a hardcoded string. In MVP, mode = 'navigation' and scene_description is omitted from the JSON schema. When scene description is enabled, mode = 'describe' adds the field back. The prompt builder is the only code that changes.

The JSON schema for scene description mode:

{ "obstacles": [ ... ], "scene_description": string, // added when mode = 'describe' "navigation_direction": string, "goal_found": boolean, "goal_confidence": number }

The app state must include a mode field ('navigation' | 'describe') initialized to 'navigation'. A future UI toggle or voice command ("describe my surroundings") sets it to 'describe' for one cycle, then returns to 'navigation'.

Configurable TTS Voice and Speed

TTS parameters (voice, rate, pitch, volume) are not user-configurable in MVP. They are hardcoded as named constants at the top of SpeechQueue, not inline literals. This is the only architectural requirement.

Architecture requirement: Define a TTS_CONFIG object at module level in SpeechQueue. All SpeechSynthesisUtterance instances read from it. When a settings UI is added, it writes to this object. No other code changes.
// SpeechQueue.js — top of file const TTS_CONFIG = { rate: 1.15, pitch: 1.0, volume: 1.0, voiceLang: 'en-US', preferLocalVoice: true }; // All utterances use this config — never hardcode values elsewhere function createUtterance(text) { const u = new SpeechSynthesisUtterance(text); u.rate = TTS_CONFIG.rate; u.pitch = TTS_CONFIG.pitch; u.volume = TTS_CONFIG.volume; u.voice = selectVoice(TTS_CONFIG.voiceLang, TTS_CONFIG.preferLocalVoice); return u; }

Goal Detection Reliability

Goal detection relies on Claude's self-reported goal_confidence value, which is unreliable for subtle visual cues — particularly small text on angled or poorly lit door signs (e.g. "room 204"). The 2-frame arrival confirmation partially mitigates this but does not solve it.

Demo route constraint: Week 4 prompt tuning must select demo destinations that are high-contrast and visually distinct: elevator doors, illuminated exit signs, large room number plaques. Avoid destinations identifiable only by small text at an angle.

Post-MVP: When goal_confidence is between 0.5 and 0.8, trigger a dedicated OCR pass on the frame rather than relying on the vision model's general confidence estimate.

Real Building Prompt Tuning (OQ-01 resolved)

Demo environment is confirmed as a real building. Prompt tuning in Week 4 must prioritize: corridor and doorway detection accuracy, sign reading in varying lighting, and distinguishing navigable space from furniture. Test scenarios to cover: hospital corridor, office hallway with multiple doors, lobby with mixed obstacles.

API Budget (OQ-04 resolved)

Budget is $5.00/day. At ~$0.003/call (claude-sonnet-4-6 with a 640×480 JPEG at 1 fps), $5 covers ~1,666 calls — approximately 27 minutes of continuous active navigation. Sufficient for demo day with headroom. Monitor via the Anthropic dashboard. No in-app enforcement needed for a single controlled device.

On-Device CV Layer (Spec 12)

Spec 12 adds MediaPipe Object Detection running at ~15fps as a parallel fast layer alongside Claude's existing 1fps vision loop. This addresses the safety latency gap: Claude's 1s cadence cannot catch a person walking toward the user within a single frame.

Design principle: CV augments Claude — it does not replace it. Claude continues to receive raw camera frames and handles all navigation reasoning, goal recognition, and spatial understanding. CV adds continuous scene awareness between Claude calls.

Architecture: three loops

LoopRateJob
Fast (rAF)~15fpscvDetector → cvTracker → cvWorldModel.update()
Medium (333ms)~3fpshazardEvaluator → estimateRisk() → obstacles.js → audio
Slow (1000ms)~1fpscvContextBuilder → Claude prompt (existing loop)
Figure 3 — Spec 12 CV layer: one fast rAF writer updates an atomic world-model snapshot; the throttled medium and slow loops only read it (skipping stale snapshots). Only the slow loop's compact CV context reaches Claude (FR-CV-03).

New requirements

FR-CV-01Parallel on-device detectionSpec 12
MediaPipe EfficientDet-Lite0 runs at ~15fps sharing the existing camera stream. It does not block or delay the Claude vision loop. Start/stop is tied to navigation start/stop.
FR-CV-02Real-time hazard alertsSpec 12
When CV detects a high-risk object (person close ahead, large obstacle in center), a brief audio alert fires via the existing obstacle speak queue within ~333ms. The alert does not interrupt active Claude navigation speech — it waits for a 2s gap. Per-class 3s cooldown prevents repetition.
FR-CV-03Claude prompt enrichmentSpec 12
On each 1fps Claude tick, a compact CV scene snapshot is prepended to the prompt when qualifying objects (medium/high risk, max 3) are currently detected and the snapshot is fresh (<1s old). Example: "CV: person center (high risk), chair right (medium risk)". When no qualifying objects, Claude prompt is sent without the prefix.
FR-CV-04Latency contractSpec 12
All consumers of the world model call cvWorldModel.getSnapshot() — the most recently completed tracker output. World model replaces its snapshot atomically (replace-not-mutate). No consumer holds a reference across ticks. Both medium and slow loops skip the snapshot if it is older than CV_CONTEXT_STALENESS_MS (1000ms).
